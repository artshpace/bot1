## ГЛАВНЫЙ ВЫВОД (ответ на критический вопрос Overview)

> **Использует ли проект реальную БД или значительная часть работает через localStorage/sessionStorage?**

**Проект ПОЛНОСТЬЮ работает на `localStorage`. Реальной БД и сервера НЕТ.**

`website/js/api.js` — это mock-CRM на ~3760 строк, где `window.API.*` читает/пишет ключи `sas_*` в `localStorage`. `backend/` — это **только скаффолд** (по его собственному README: *"The frontend runs as a pure static site with a localStorage mock API. These files show exactly what the backend needs to implement"*). Ни один backend-эндпоинт не развёрнут и не вызывается.

**Следствие:** почти все требования спек 1.1.1 (реальная БД, JWT в http-only cookie, bcrypt, серверная авторизация, SMTP, реальный Telegram-бот, платёжный шлюз) **не выполнены by design** — это прототип, а не продакшн-бэкенд. Объём работ = построение настоящего backend с нуля.

## ВЫВОДЫ АУДИТА

### Главный вопрос
**localStorage.** Реальной БД/сервера нет. `api.js` — mock на localStorage; `backend/` — недеплоенный скаффолд.

### Что оказалось заглушкой
- `auth.recover()` — возвращает `{ok:true}`, ничего не отправляет (api.js:910)
- Telegram-уведомления — только очередь `sas_tg_outbox`, фактической отправки нет
- Backend webhooks (kaspi/freedompay/cloudpayments/telegram) — скаффолд-файлы, не работают
- Email-подсистема — отсутствует полностью

### Что использует localStorage
Всё: `sas_users, sas_session, sas_cart, sas_orders, sas_subscriptions, sas_courses, sas_schedule_slots, sas_journal, sas_notifications, sas_tg_*` и др. (38 ключей)

### Что использует временные данные
Все сущности инициализируются из SEED-блоков `api.js` через `ensureSeed()` (6 юзеров, абонементы, курсы, расписание, преподаватели, сертификаты).

### Критические проблемы (по приоритету)
1. **Пароли в открытом виде** в localStorage (XII) — нет хеширования
2. **Нет реального backend/БД** — вся «авторизация» клиентская, роль подменяется правкой localStorage (II, IV, XII)
3. **Нет подтверждения email** и автовход после регистрации (II, III, V) — прямое нарушение SPEC_AUTH §1
4. **Восстановление пароля не работает** (заглушка) (V, XIII)
5. **Глобальная корзина** не привязана к user_id — утечка между аккаунтами (VII)
6. **Нет rate limiting / журнала входов / серверной проверки прав** (XII)
7. **Telegram**: фактической отправки/приёма нет; username бота `shpigotskiy_art_bot` ≠ спека `@artshpacebot`
8. **Каталог** без схемы статусов active/inactive/draft и UUID-структуры из SPEC_SHOP §2
9. **IDOR**: `orders.get(id)` не проверяет владельца

### Технический долг
Чтобы выполнить спеки 1.1.1, нужен **реальный backend** (Node/Express + БД + ORM), которого сейчас нет:
- БД (PostgreSQL/MySQL) + ORM (Prisma/Sequelize) для users/orders/cart/subscriptions/tokens/logs
- Auth: bcrypt(≥12), JWT в http-only secure cookie, refresh-токены, email-подтверждение, рабочее восстановление с TTL-токенами
- SMTP-провайдер (письма подтверждения/восстановления/покупок)
- Telegram-бот: вебхук, обработка `/start`,`/link`, сохранение chat_id, реальная отправка уведомлений
- Платёжный шлюз: серверные create + webhook-фулфилмент (скаффолды в `backend/` уже намечают это)
- Серверный энфорсмент прав на каждом эндпоинте, CSRF, rate limiting, login logs, HTTPS/HSTS

### Что УЖЕ сделано хорошо (фундамент для миграции)
- Чистая модель данных и API-контракт (`window.API.*`) — переносится на реальный backend почти 1:1
- Pay-before-access для заказов (`awaiting_payment` → `paid` → fulfil) реализован
- Заказы привязаны к `userId`, история фильтруется по пользователю
- Telegram/платежи спроектированы безопасно (токен не в браузере), `backend/` скаффолд намечает серверную часть
- Ролевое разделение UI готово (`auth.js`) — останется добавить серверную проверку
