# VERSION 1.1.1 — ИНСТРУКЦИИ ПО ИСПОЛЬЗОВАНИЮ СПЕЦИФИКАЦИИ

## Структура файлов

```
SPEC_1.1.1_OVERVIEW.md          — Обзор версии 1.1.1 и основные принципы
AUDIT_CHECKLIST.md              — Полный чеклист технического аудита
SPEC_AUTH.md                    — Требования: Авторизация и аккаунты
SPEC_SHOP.md                    — Требования: Магазин и покупки
SPEC_TELEGRAM_SECURITY.md       — Требования: Telegram и безопасность
README_1.1.1.md                 — Этот файл
```

## Как использовать эти файлы

### Шаг 1: Проведение технического аудита

1. Открыть `AUDIT_CHECKLIST.md`
2. Для каждого раздела проверить фактическую реализацию
3. Заполнить результаты в чеклисте
4. **Не делать предположений** — только факты

**Критический вопрос перед началом:**

> Использует ли проект реальную базу данных пользователей или значительная часть логики всё ещё работает через localStorage/sessionStorage?

**Ответ на этот вопрос определяет объём всех работ.**

### Шаг 2: Анализ результатов аудита

На основе результатов аудита определить:

- Что оказалось заглушкой (mock функции)
- Что использовало localStorage (вместо БД)
- Что использовало временные данные
- Какие критические проблемы безопасности
- Какой технический долг

### Шаг 3: Планирование работ

На основе результатов составить план исправлений:

1. **Критические** (безопасность, утечка данных) → первый приоритет
2. **Важные** (авторизация, платежи) → второй приоритет
3. **Желательные** (удобство, оптимизация) → третий приоритет

### Шаг 4: Разработка по спецификациям

Для каждой компоненты использовать соответствующую спецификацию:

- **SPEC_AUTH.md** — разработка авторизации
- **SPEC_SHOP.md** — разработка магазина и покупок
- **SPEC_TELEGRAM_SECURITY.md** — Telegram и безопасность

## Основные проверки перед началом

### Для авторизации (SPEC_AUTH.md)

- [ ] Пользователи хранятся в реальной БД?
- [ ] После регистрации требуется подтверждение email?
- [ ] Пароли хешируются (bcrypt/argon2)?
- [ ] Есть защита от перебора?
- [ ] Сессия сохраняется после обновления страницы?

### Для магазина (SPEC_SHOP.md)

- [ ] Существует единый каталог абонементов?
- [ ] Корзина привязана к пользователю (не общая)?
- [ ] Заказы сохраняются в БД?
- [ ] Покупка требует реальную оплату?
- [ ] Новый аккаунт создаётся без тестовых данных?

### Для Telegram (SPEC_TELEGRAM_SECURITY.md)

- [ ] Используется бот @artshpacebot?
- [ ] Привязка требует код подтверждения?
- [ ] Сохраняется chat_id привязанного пользователя?
- [ ] Отправляются ли реальные уведомления?

## Важные принципы разработки

### ❌ Запрещено

- Создавать заглушки (mock функции) на продакшене
- Имитировать работу функций
- Хранить критические данные в localStorage
- Проверять права доступа только на клиенте
- Хранить пароли открытым текстом
- Отправлять письма в консоль вместо SMTP

### ✓ Требуется

- Реальная база данных для хранения пользователей
- Реальная авторизация с JWT/session
- Реальная отправка писем через SMTP
- Реальная интеграция Telegram
- Проверка прав на сервере
- Хеширование паролей (bcrypt/argon2)
- HTTPS для всех соединений
- Rate limiting против перебора

## Критические вопросы для новой разработки

### Перед исправлением авторизации

1. Где сейчас хранятся пользователи? (БД, localStorage, memory)
2. Используются ли реальные JWT токены или заглушка?
3. Может ли пользователь изменить роль через браузер?
4. Используется ли HTTPS или HTTP?
5. Какой SMTP используется для писем?

### Перед исправлением магазина

1. Все товары в единой БД или разрозненные?
2. Корзина пользователя или общая для всех?
3. Какой платёжный шлюз используется?
4. Есть ли реальная обработка платежей или заглушка?
5. Создаются ли реальные заказы в БД?

### Перед исправлением Telegram

1. Существует ли реальный Telegram бот @artshpacebot?
2. Какой механизм привязки используется?
3. Сохраняется ли chat_id в БД?
4. Отправляются ли реальные уведомления через API бота?

## Финальный отчёт

После завершения всех работ предоставить отчёт с:

### Для каждого пункта спецификации

- ✓ Подтверждено (исправлено)
- ✗ Не подтверждено (не требует исправления)
- ⚠ Исправлено (что именно)
- ❌ Требует доработки (что и почему)

### Разделы отчёта

1. **Что оказалось заглушкой** — полный список mock функций
2. **Что использовало localStorage** — что было перенесено на БД
3. **Что использовало временные данные** — что было исправлено
4. **Что было исправлено** — подробный список изменений
5. **Что осталось техническим долгом** — что требует доработки
6. **Главный вопрос** — ответ о реальности БД

### Должен содержать

- Не скрывать проблемы
- Не маскировать заглушки под рабочий функционал
- Если что-то требует внешних сервисов (SMTP, Telegram, платёжи), указать это
- Привести список всех API ключей и настроек, которые требуют конфигурации

## Где искать информацию

### О текущей реализации

- `index.html` — главный файл приложения (смотреть JavaScript)
- `.git/logs` — история коммитов
- Консоль браузера (DevTools) — что хранится в localStorage
- Сетевые запросы (Network tab) — какие эндпойнты вызываются
- Исходный код фронтенда и бэкенда

### Где хранятся тестовые данные

- В коде (hardcoded переменные с test, demo, mock в имени)
- В localStorage (console: localStorage.getItem('...'))
- В БД (SELECT * FROM users WHERE name LIKE '%test%')
- В миграциях или seed-файлах
