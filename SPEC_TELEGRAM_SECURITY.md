# SPEC 1.1.1 — TELEGRAM И БЕЗОПАСНОСТЬ

## 1. TELEGRAM ИНТЕГРАЦИЯ

### Текущая проблема

- Привязка Telegram может быть неправильной
- Коды подтверждения могут не обрабатываться
- Бот может быть неправильно конфигурирован

### Требования к боту

- [ ] Username бота: `@artshpacebot`
- [ ] Проверить название бота в Telegram
- [ ] Проверить описание бота
- [ ] Проверить, получает ли бот сообщения
- [ ] Проверить, обрабатываются ли команды

### Команды бота

- `/start` — Начало диалога, предложение привязки
- `/link [код]` — Привязка аккаунта (с кодом)
- `/profile` — Показать информацию профиля
- `/schedule` — Показать расписание занятий
- `/balance` — Показать баланс абонементов

### Информация, отправляемая в Telegram

Когда пользователь нажимает "Привязать Telegram":
- [ ] На сайте генерируется уникальный код (например, `TG-ABC123-XYZ`)
- [ ] Код отображается пользователю
- [ ] Код имеет срок действия (например, 10 минут)
- [ ] Пользователю предлагается отправить код боту

Когда пользователь отправляет код боту:
- [ ] Бот получает сообщение с кодом
- [ ] Бот проверяет валидность кода
- [ ] Бот определяет `chat_id` отправителя
- [ ] Бот привязывает `chat_id` к аккаунту
- [ ] Бот отправляет подтверждение пользователю
- [ ] На сайте появляется сообщение об успешной привязке

## 2. ПРИВЯЗКА TELEGRAM

### Требования

- [ ] Существует реальная интеграция Telegram
- [ ] Бот получает сообщения от пользователей
- [ ] Коды подтверждения сохраняются в БД
- [ ] chat_id сохраняется в профиле пользователя
- [ ] Запрещена фиктивная привязка (заглушка)

### Процесс привязки

1. Пользователь нажимает "Привязать Telegram" в ЛК
2. Система генерирует уникальный код (UUID, первые 8 символов)
3. Код сохраняется в БД с TTL 10 минут
4. Пользователю показывается инструкция с кодом
5. Пользователь открывает бот (@artshpacebot)
6. Пользователь отправляет команду `/link TG-ABC123-XYZ`
7. Бот проверяет код в БД
8. Бот получает chat_id и user_id из очереди/базы
9. Бот сохраняет chat_id в таблице пользователей
10. Бот отправляет подтверждение
11. На сайте появляется сообщение об успехе

### Структура привязки

```
user_table:
{
  id: string
  ...
  telegram_chat_id: string | null
  telegram_linked_at: timestamp | null
}

telegram_confirmation_code:
{
  code: string (unique)
  user_id: string
  created_at: timestamp
  expires_at: timestamp
  used: boolean
}
```

## 4. БЕЗОПАСНОСТЬ: ХЕШИРОВАНИЕ ПАРОЛЕЙ

### Текущая проблема

- [ ] Хранятся ли пароли открытым текстом?
- [ ] Используются ли слабые хеши?

### Требования

- [ ] Пароли должны хешироваться (bcrypt, argon2, scrypt)
- [ ] Минимум 10 раундов для bcrypt
- [ ] Минимум 12 раундов рекомендуется для bcrypt
- [ ] Никогда не хранить пароли открытым текстом
- [ ] Никогда не отправлять пароли в письмах
- [ ] При восстановлении пароля отправлять ссылку, а не пароль

### Реализация

```javascript
// Правильно
const hashedPassword = await bcrypt.hash(password, 12);
const isValid = await bcrypt.compare(inputPassword, hashedPassword);

// Неправильно ❌
const user = { password: password }; // OPEN TEXT
```

## 5. БЕЗОПАСНОСТЬ: ЗАЩИТА ОТ ПЕРЕБОРА

### Требования

- [ ] Максимум 5 попыток входа за 5 минут
- [ ] После превышения заблокировать аккаунт на время (15 минут)
- [ ] Пользователю показать сообщение об блокировке
- [ ] Предложить восстановление пароля
- [ ] Логировать все попытки входа

### Реализация

```
login_attempts:
{
  user_id_or_email: string
  attempt_count: number
  first_attempt_at: timestamp
  last_attempt_at: timestamp
  locked_until: timestamp | null
}
```

## 7. БЕЗОПАСНОСТЬ: ХРАНЕНИЕ ТОКЕНОВ

### JWT токены

- [ ] Сохранять в http-only cookie (не в localStorage)
- [ ] Сохранять в secure cookie (только HTTPS)
- [ ] Сохранять SameSite flag (Strict или Lax)
- [ ] Имеет короткий TTL (15 минут)
- [ ] Есть refresh token с длинным TTL (7 дней)

### Реализация

```javascript
res.cookie('accessToken', token, {
  httpOnly: true,
  secure: true,          // Только HTTPS
  sameSite: 'Strict',
  maxAge: 15 * 60 * 1000 // 15 минут
});
```

## 8. БЕЗОПАСНОСТЬ: ЗАЩИТА ПРАВ ДОСТУПА

### Требования

- [ ] Все права проверяются на сервере
- [ ] Клиент скрывает элементы UX, но сервер отклоняет запросы
- [ ] Невозможно подменить роль через URL
- [ ] Невозможно подменить роль через cookies/localStorage
- [ ] Невозможно получить доступ к чужому ЛК через URL `/account/user-id/123`
- [ ] Невозможно получить доступ к админке без прав

### Пример: Защита эндпойнта

```javascript
// ❌ НЕПРАВИЛЬНО - только клиентская проверка
if (user.role === 'admin') {
  // Показать админку
}

// ✓ ПРАВИЛЬНО - проверка на сервере
GET /api/admin/users
  Authorization: Bearer {token}
  
  // На сервере:
  const user = verifyToken(token);
  if (user.role !== 'admin') {
    return 403 Forbidden;
  }
```

## 9. БЕЗОПАСНОСТЬ: ЗАЩИТА ДАННЫХ ПОЛЬЗОВАТЕЛЯ

### Требования

- [ ] Пользователь не может видеть данные другого пользователя
- [ ] API фильтрует данные по user_id из токена
- [ ] Нельзя получить данные другого пользователя через `/api/user/123`
- [ ] Нельзя получить корзину другого пользователя
- [ ] Нельзя получить заказы другого пользователя
- [ ] Нельзя получить расписание другого пользователя

### Пример: Защита данных

```javascript
// ❌ НЕПРАВИЛЬНО
GET /api/user/123 → возвращает данные пользователя 123

// ✓ ПРАВИЛЬНО
GET /api/user/me → возвращает данные текущего пользователя
// Получаемое из токена: user_id = 123
```

## 11. БЕЗОПАСНОСТЬ: SQL INJECTION И ДРУГОЕ

### Требования

- [ ] Использовать параметризованные запросы (не конкатенация)
- [ ] Использовать ORM (Sequelize, TypeORM, Prisma)
- [ ] Валидировать входные данные на сервере
- [ ] Экранировать выходные данные

### Пример

```javascript
// ❌ УЯЗВИМО
const user = await db.query(`SELECT * FROM users WHERE email = '${email}'`);

// ✓ БЕЗОПАСНО
const user = await db.query('SELECT * FROM users WHERE email = ?', [email]);
```

## 12. HTTPS И БЕЗОПАСНОСТЬ СОЕДИНЕНИЯ

### Требования

- [ ] Все эндпойнты на HTTPS (не HTTP)
- [ ] Редирект с HTTP на HTTPS
- [ ] HSTS заголовок (Strict-Transport-Security)
- [ ] SSL сертификат валидный и не истекший
- [ ] TLS 1.2 и выше

### HSTS заголовок

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

### На сервере

- [ ] Проверять диапазон значений
