## P1 — Задача 3: Письма Gmail не должны уходить в спам

### Цель
Письма Supabase (подтверждение почты, сброс пароля) приходят во «Входящие», а не в «Спам».

### Суть проблемы
Сейчас (для теста) используется Gmail SMTP от лица `artshpace@gmail.com`. Gmail как SMTP-релей для
транзакционных писем имеет плохую доставляемость и часто помечается спамом, т.к. домен отправителя
не совпадает с инфраструктурой и нет настроенных SPF/DKIM/DMARC под рассыльщика.

### Рекомендуемое решение (по возрастанию усилий)
1. **Перейти на транзакционный SMTP-провайдер** (бесплатные тарифы): Resend, Brevo (Sendinblue),
   Mailgun, Postmark. В Supabase → Authentication → Emails → SMTP Settings вписать их хост/порт/логин/ключ.
   Это сразу резко улучшает доставляемость даже без своего домена (провайдер шлёт со своего проверенного домена).
2. **Свой домен + DNS-записи** (когда купите домен вместо github.io):
   - SPF: `TXT @ "v=spf1 include:<провайдер> ~all"`
   - DKIM: CNAME/TXT-записи, которые выдаёт провайдер.
   - DMARC: `TXT _dmarc "v=DMARC1; p=none; rua=mailto:artshpace@gmail.com"`
   - В Supabase указать `From` на адрес вашего домена (напр. `noreply@вашдомен`).
3. **Тексты писем:** в Supabase → Authentication → Emails → шаблоны. Указать имя отправителя
   «Shpigotskiy Art Space», убрать «голый» технический текст — это снижает спам-рейтинг.

### Действия владельца
- Зарегистрироваться у одного провайдера (рекомендация: **Resend** — простой, щедрый free tier).
- Получить SMTP-логин и API-ключ, вписать в Supabase. Ключ НЕ присылать в чат — вводить напрямую в Supabase.
- (Опц.) при покупке домена — добавить SPF/DKIM/DMARC по инструкциям провайдера.

### Критерии приёмки
- Тестовая регистрация нового аккаунта → письмо подтверждения во «Входящих» Gmail/Mail.ru, не в спаме.
- Заголовок «От кого» читается как «Shpigotskiy Art Space», а не как сырой адрес.

### Роль Claude
Кодовых изменений почти нет — это конфигурация Supabase + DNS. Claude: подготовить точные значения
DNS-записей под выбранного провайдера и проверить шаблоны писем. Документировать в этом разделе фактические значения после настройки.

## P1 — Задача 4: Привязка Telegram-бота к аккаунту на сайте (уведомления)

### Цель
Пользователь в ЛК нажимает «Подключить Telegram», получает код/ссылку, активирует бота
**@artshpacebot**, и его Telegram-аккаунт привязывается к аккаунту на сайте. После этого бот шлёт
ему персональные уведомления (напоминания о занятии, статус заявки и т.п.).

### Что уже готово
- Таблица `public.telegram_codes (code, user_id, used, created_at, expires_at)` — `0001_init.sql:226`.
  RLS: пользователь видит только свои коды. Код живёт 10 минут.
- Worker-инфраструктура (`workers/lead-forwarder.js`) — образец того, как Worker ходит в Telegram API.

### Архитектура (deep-link binding)
1. **Сайт (ЛК):** кнопка «Подключить Telegram» →
   - генерит случайный `code` (напр. 8 символов), пишет в `telegram_codes` через supa.js:
     `client.from('telegram_codes').insert({ code, user_id })`.
   - открывает `https://t.me/artshpacebot?start=<code>` (deep-link: Telegram передаст код боту как `/start <code>`).
2. **Бот (нужен второй Worker или хостинг бота):** на `/start <code>` бот:
   - дёргает Supabase REST (service-role ключ, в env Worker) → ищет `telegram_codes` где `code=<code>`,
     `used=false`, `expires_at > now()`.
   - если найден — пишет `telegram_chat_id` в `profiles` этого `user_id` и помечает код `used=true`.
   - отвечает в чат: «✅ Аккаунт привязан. Теперь вы будете получать уведомления».
3. **Схема:** добавить колонку `profiles.telegram_chat_id text` (миграция `0004_telegram_link.sql`),
   и индекс. Туда бот пишет chat_id.
4. **Отправка уведомлений:** существующий/новый Worker по `telegram_chat_id` шлёт `sendMessage`.

### Затрагиваемые файлы
- `supabase/migrations/0004_telegram_link.sql` (СОЗДАТЬ): `alter table profiles add column telegram_chat_id text;`
  + RLS: пользователь видит свой, director/admin — все. Бот пишет через service-role (минует RLS).
- `workers/telegram-bot.js` (СОЗДАТЬ): обработчик вебхука бота (`/start <code>` → привязка).
  Env: `TELEGRAM_BOT_TOKEN`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`.
- `website/js/account.js` ИЛИ новый `website/js/telegram-link.js` — кнопка «Подключить Telegram» в settings.html.
  Предпочтительно отдельный модуль, подключаемый на `settings.html`, чтобы не трогать account.js.

### Действия владельца (ИНСТРУКЦИЯ — то, что Антон спрашивал)
1. У **@BotFather** убедиться, что бот @artshpacebot существует, взять его токен (уже есть из задачи Worker).
2. **Включить deep-linking** ничего настраивать не надо — `?start=` работает у всех ботов из коробки.
3. **Зарегистрировать вебхук бота** один раз (после деплоя `telegram-bot.js`):
   `https://api.telegram.org/bot<TOKEN>/setWebhook?url=https://<новый-worker>.workers.dev`
   (Claude даст готовую команду с подставленным URL Worker; токен Антон подставляет сам.)
4. В Cloudflare добавить в env нового Worker: `TELEGRAM_BOT_TOKEN`, `SUPABASE_URL`,
   `SUPABASE_SERVICE_ROLE_KEY` (Secret key из Supabase → Settings → API). Service-role ключ — СЕКРЕТ,

5. Прислать Claude URL нового Worker — Claude впишет deep-link и команду setWebhook.

### Критерии приёмки
- В `settings.html` кнопка «Подключить Telegram» открывает бота с предзаполненным `/start <код>`.
- После нажатия Start в боте — приходит «✅ Аккаунт привязан», в `profiles.telegram_chat_id` появился chat_id.
- Повторный/просроченный код не срабатывает (бот отвечает «код недействителен»).

### Риски

- Бот может писать пользователю только если тот раньше нажал Start — это и обеспечивается потоком.

## P1 — Задача 7: Вход по ФИО (помимо email/телефона)

### Цель
Пользователь может войти, введя ФИО (на которое зарегистрирован аккаунт) + пароль, не только email.

### Сложность
Supabase Auth логинит строго по email (или телефону). По «ФИО» напрямую он не умеет. Нужен
промежуточный поиск: ФИО → email → стандартный вход по email.

### Решение
1. На форме входа (`website/account/login.html`) определять: введённое значение — это email
   (содержит `@`), телефон (цифры), или ФИО (буквы с пробелом).
2. Если ФИО: в `supa.js` сделать lookup через RPC-функцию Supabase (нельзя читать чужие email напрямую —
   нужна `security definer` функция):
   - Миграция `0005_login_by_name.sql`: функция `public.email_by_name(p_name text) returns text`
     `security definer` — ищет в `profiles` по `name` (точное или нормализованное совпадение),
     джойнит `auth.users` за email. Возвращает email ТОЛЬКО если ФИО уникально (иначе null → просим уточнить).
   - **Безопасность:** функция не должна позволять перебор. Вернуть email лишь при точном уникальном
     совпадении ФИО; при неоднозначности — общая ошибка «уточните email». Рассмотреть rate-limit.
3. `supa.js signIn()`: если ввели ФИО → сперва `rpc('email_by_name')`, затем обычный
   `signInWithPassword({ email, password })`.
4. Подсказка на форме: «Email, телефон или ФИО».

### Затрагиваемые файлы
- `supabase/migrations/0005_login_by_name.sql` (СОЗДАТЬ)
- `website/js/supa.js` — ветвление в `signIn()`
- `website/account/login.html` — placeholder/label

### Критерии приёмки
- Вход по «Иванов Иван Иванович» + пароль работает, если ФИО уникально.
- При совпадающих ФИО у разных людей — просьба войти по email (без утечки, какой именно email существует).
- Email/телефон-вход продолжает работать как раньше.

### Риски / ограничения
- ФИО не уникально по природе — поэтому email остаётся основным идентификатором, ФИО лишь удобный alias.
- Не раскрывать существование аккаунта по ФИО (защита приватности).

## P2 — Задача 6: Напоминание в календарь УЧЕНИКА/РОДИТЕЛЯ (после записи)

> Делается раньше задачи 5, потому что НАМНОГО проще (без OAuth/серверного API).

### Цель
После успешной записи на пробное — предложить добавить событие в календарь телефона, чтобы не забыли.

### Решение (без серверов, чисто на клиенте)
На экране успеха формы (`#modal-form .form-success`) показать две кнопки:
1. **«Добавить в Google Календарь»** — ссылка-генератор:

2. **«Скачать .ics»** (для Apple Calendar / любого телефона) — сгенерировать ICS-файл на лету

Данные брать из формы: направление, выбранный день+слот (уже собираются в `main.js`, поле slot),
адрес — `ул. Интернациональная, 63, 5 этаж` (из CLAUDE.md).
**Тонкость:** выбранный «день недели + время» нужно превратить в конкретную дату ближайшего такого дня.
Написать хелпер `nextDateForWeekday(dayName, timeRange)` в `main.js`.

### Затрагиваемые файлы

- `website/js/main.js` — генераторы Google-ссылки и .ics, хелпер даты.

### Критерии приёмки
- После записи на гитару (напр. «Среда, 17:00–18:00») кнопка «Google Календарь» открывает черновик
  события на ближайшую среду 17:00–18:00 с названием «Пробное занятие — Гитара» и адресом студии.
- Кнопка «.ics» скачивает файл, который открывается в календаре телефона с теми же данными.

## P2 — Задача 5: Авто-напоминание в Google Календарь ДИРЕКТОРА о новых записях

> Сложнее задачи 6: пишет в ЧУЖОЙ (директорский) календарь автоматически → нужен серверный OAuth.

### Цель
Каждая новая заявка на пробное автоматически создаёт событие в Google Календаре директора:
«Пробное занятие по "<направление>", <ФИО записавшего>» с датой/временем выбранного слота.

### Архитектура
Расширить серверную часть, которая уже принимает лиды (Cloudflare Worker `lead-forwarder.js` или
новый Worker), чтобы помимо отправки в Telegram она создавала событие через **Google Calendar API**.

1. **Авторизация Google:** нужен сервисный доступ к календарю директора. Два пути:
   - **Service Account + расшаренный календарь** (рекомендуется для сервера): создать сервисный аккаунт
     в Google Cloud, включить Calendar API, расшарить календарь Антона на email сервисного аккаунта с
     правом «вносить изменения». Worker подписывает JWT → получает access_token → создаёт событие.
   - (Альтернатива — OAuth refresh-token, сложнее в поддержке.)
2. **Worker** на каждый валидный лид с распознанным слотом:
   - вычисляет дату ближайшего нужного дня/времени (тот же хелпер, что в задаче 6, но серверная копия),
   - `POST https://www.googleapis.com/calendar/v3/calendars/<calId>/events` с телом события,
   - title: `Пробное занятие — <направление>`, description: `<ФИО>, тел: <телефон>`, location: адрес студии.

### Затрагиваемые файлы
- `workers/lead-forwarder.js` (РАСШИРИТЬ) или `workers/calendar-sync.js` (СОЗДАТЬ).
- Env Worker: `GOOGLE_SA_EMAIL`, `GOOGLE_SA_PRIVATE_KEY`, `GOOGLE_CALENDAR_ID`.

### Действия владельца (Антон)
1. Google Cloud Console → создать проект → включить **Google Calendar API**.
2. Создать **Service Account**, выпустить JSON-ключ (private key). Ключ — СЕКРЕТ, вводить только в
   Cloudflare как Secret, не в чат.
3. В Google Календаре → Настройки нужного календаря → «Доступ для отдельных пользователей» →
   добавить email сервисного аккаунта с правом «Внесение изменений в мероприятия».
