## P0 — Задача 10: Электронный журнал с посещаемостью (фундамент данных)

### Цель
Журнал, где преподаватель/админ ведёт не только темы и список учеников, но и **отметки посещаемости
по датам согласно расписанию** — как школьный классный журнал (строки = ученики, столбцы = даты занятий).

### Почему первая
Посещаемость — базовые данные, на которых стоят задачи 8 (перерасчёт по фактическим занятиям) и
9 (карта развития/оценки). Без неё они работают на «воздухе».

### Модель данных (миграция, напр. `0010_journal.sql`)
- `groups` (id, direction, name, teacher_id, schedule jsonb — дни/время из реального расписания студии).
- `group_members` (group_id, student_id) — кто в группе.
- `lessons` (id, group_id, date, topic, status) — конкретное занятие на дату (генерится из расписания группы).
- `attendance` (lesson_id, student_id, status) — статус: `present | absent | excused | sick` (+ ссылка на справку).
- RLS: преподаватель видит/правит свои группы; admin/director — все; ученик/родитель — только свои отметки.

### Расписание-генератор
Дни/слоты направлений уже зашиты в `website/js/main.js` → `const SCHEDULE` (гитара Пн/Ср/Пт + слоты;
вокал/живопись Сб/Вс). Переиспользовать ту же истину при генерации `lessons` группы на период.

### UI
- `admin-journal.html` / `journal.html`: сетка «ученики × даты». Клик по ячейке — циклическая смена
  статуса (был/не был/уважит./болел). Шапка столбца — дата занятия. Итоги по строке (посещено N из M).
- Фильтры: направление, группа, месяц.
- Преподаватель видит только свои группы (по `teacher_id = auth.uid()`).

### Затрагиваемые файлы
- `supabase/migrations/0010_journal.sql` (СОЗДАТЬ)
- `website/account/admin-journal.html`, `website/account/journal.html` (ДОРАБОТАТЬ)
- `website/js/api.js` — добавить методы `API.journal.*` ИЛИ читать Supabase напрямую через supa.js.
  Решить: мок или реальная БД. Для продакшена — Supabase; для демо можно оставить мок-ветку.
- Новый `website/js/journal.js` (вынести логику сетки, чтобы не раздувать account.js).

### Критерии приёмки
- Преподаватель открывает свою группу, видит сетку учеников × дат месяца, проставляет посещаемость кликом.
- Итог «посещено 6 из 8» считается автоматически.
- Ученик/родитель видит только свои отметки, не чужие.

### Риски
- Не сломать существующий рендер journal.html. Изменения изолировать в journal.js.

## P1 — Задача 8: Фактический перерасчёт абонемента в кабинете админа/директора

### Цель
Красивая таблица/калькулятор автоматического перерасчёта стоимости исходя из: фактически посещённых
занятий, загруженных справок/документов, скидок (если были) и цены абонемента по конкретному направлению.

### Зависимость
Нужны данные посещаемости (Задача 10) и абонементов. Без задачи 10 перерасчёт неоткуда брать «факт».

### Логика
- Вход: ученик + период (месяц) + направление.
- Берём: оплачено занятий по абонементу (план), фактически посещено (из `attendance`), пропуски по
  болезни со справкой (`status='sick'` + документ) → подлежат перерасчёту; пропуски без причины — нет.
- Формула: `к возврату/переносу = цена_занятия × (засчитанные_пропуски)`, с учётом скидки.
- Цена занятия = цена абонемента направления / число занятий в абонементе (из настроек директора,
  `sas_director_pricing` в localStorage или таблицы `products`/`pricing` в Supabase).

### UI
`admin-recalculations.html` (каркас есть): таблица — Ученик · Направление · План · Факт · Болезнь(справка)
· Скидка · К перерасчёту · [Применить]. Кнопка «Применить» фиксирует перерасчёт (запись в БД/историю).
Доступ: только `admin`/`director`.

### Затрагиваемые файлы
- `website/account/admin-recalculations.html` (ДОРАБОТАТЬ)
- `supabase/migrations/00xx_recalculations.sql` (таблица истории перерасчётов, если нужна фиксация)
- `website/js/api.js` или новый `website/js/recalc.js` — расчётная логика.

### Критерии приёмки
- Для ученика с 2 пропусками по болезни (со справкой) при цене занятия X — таблица показывает к
  перерасчёту 2X с учётом скидки; пропуски без справки не учитываются.
- Перерасчёт можно «Применить» и увидеть в истории.

### Риски / ограничения
- Правила перерасчёта согласовать с Антоном (политика студии: справка в течение 7 дней — см. FAQ на сайте).
- Деньги — чувствительно: показывать формулу прозрачно, ничего не списывать автоматически без подтверждения.

## P1 — Задача 9: Карта развития и оценочная система (видят только 18+ ученики и родители ребёнка)

### Цель
Продуманная карта развития/навыков и оценки, доступные **только**: ученикам 18+ (свои) и родителям
конкретного ребёнка (своего). Детям младше 18 оценки НЕ показывать — чтобы не создавать стресс.

### Ключевая тонкость (приватность + педагогика)
- Видимость оценок гейтится по роли И возрасту:
  - ученик с `age >= 18` → видит свою карту/оценки;
  - родитель → видит карту своего ребёнка (связь parent→child);
  - ученик `< 18` → видит карту развития в «мягком» виде (прогресс/навыки, БЕЗ числовых оценок и негатива).
- Это требует модели «навыков» и «прогресса», а не только баллов: акцент на рост, а не на оценку.

### Модель данных (миграция `00xx_skillmap.sql`)
- `skills` (id, direction, name, description) — справочник навыков по направлению.
- `student_skill_progress` (student_id, skill_id, level 0..N, updated_at, teacher_id).
- `assessments` (id, student_id, lesson_id|date, score, comment, visible_to jsonb) — оценки с правилом видимости.
- Связь родитель→ребёнок: таблица `parent_child (parent_id, child_id)` (если ещё нет — добавить).
- RLS: жёстко по правилам выше. Возраст ученика хранить в `profiles` (добавить `birth_date` или `age`).

### UI
- `admin-skillmap.html` (каркас есть) — преподаватель/админ выставляет прогресс по навыкам и оценки.
- В кабинете ученика/родителя — карта развития: радиальная/столбчатая визуализация навыков; оценки
  показываются ТОЛЬКО при выполнении правил видимости (иначе скрыты, без следов).
- Для детей <18: режим «карта роста» — навыки и достижения, без чисел и красных пометок.

### Затрагиваемые файлы
- `supabase/migrations/00xx_skillmap.sql` (СОЗДАТЬ)
- `website/account/admin-skillmap.html` (ДОРАБОТАТЬ)
- Кабинет ученика/родителя — раздел «Карта развития» (новый модуль `website/js/skillmap.js`).

### Критерии приёмки
- Родитель видит карту своего ребёнка с оценками; чужого ребёнка — не видит (RLS).
- Ученик 18+ видит свои оценки; аккаунт ребёнка <18 видит только «мягкую» карту без числовых оценок.
- Преподаватель может выставлять навыки/оценки только своим ученикам.

### Риски
- Это самая деликатная задача по приватности — RLS проверить особенно тщательно (тест от лица каждой роли).
- Педагогическую модель «мягкой» карты согласовать с Антоном (что именно показывать детям).

## P2 — Задача 11: Воронка продаж на фактических данных

### Цель
Воронка показывает реальные числа из источников трафика (таргет/Google/прочее), а не демо-моки:
показы/клики → лиды → записи на пробное → дошли → купили абонемент.

### Зависимость
Реальные лиды уже частично текут (форма → Telegram + CRM-лид в api.js). Нужно: (а) сохранять лиды и
их статусы в Supabase, (б) подтягивать данные рекламных кабинетов.

### Этапы
1. **Лиды в БД:** таблица `leads` в Supabase (name, phone, direction, slot, source, utm_*, status,
   created_at). `main.js submitLead()` (или Worker) пишет туда. Статусы: `new → contacted → trial_booked
   → trial_attended → purchased → lost`. Менять статус в `admin-leads.html`.
2. **Воронка из БД:** `admin-funnel.html` считает конверсии по реальным статусам лидов за период.
3. **Источники трафика (показы/клики):**
   - **UTM уже собираются** в `main.js` (getUTM). Источник лида известен.
   - Для показов/кликов из рекламы: интеграция с Meta Ads / Google Ads API ИЛИ ручной/CSV-импорт
     расходов и показов (проще на старте). Решить с Антоном: API-интеграция дороже, CSV — быстрее.
4. **Метрика:** подключить Яндекс.Метрику/GA для трафика сайта (отдельная быстрая задача из общего плана).

### Затрагиваемые файлы
- `supabase/migrations/00xx_leads.sql` (СОЗДАТЬ): таблица leads + RLS (staff видит все; запись с сайта —
  через Worker service-role или anon-insert с ограничениями).
- `website/account/admin-leads.html`, `website/account/admin-funnel.html`, `website/account/admin-ads.html`
  (ДОРАБОТАТЬ — читать реальные данные).
- `website/js/main.js` или Worker — запись лида в Supabase.

### Критерии приёмки
- Новая заявка с сайта появляется в `admin-leads.html` с источником/UTM.
- В `admin-funnel.html` конверсии считаются по фактическим статусам лидов, а не по демо-данным.
- Видна разбивка по источникам (Instagram/таргет/сарафан/прочее) на основе UTM.

### Риски
- Реклама-API (Meta/Google) — сложные и требуют верификации приложения. На старте достаточно UTM +
  ручной ввод бюджета. Полную API-интеграцию выносить отдельной под-задачей.
